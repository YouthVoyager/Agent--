# 投递语义

## 非持久模式

```text
收到请求 -> 数据成功放入内存队列 -> 返回 ACK
```

这种模式延迟低，但进程崩溃时内存队列中的数据可能丢失。适合成本敏感、允许少量丢失的流量。

## 持久模式

```text
收到请求 -> 写入 WAL -> 按配置完成同步 -> 返回 ACK
```

这种模式用于关键路由。ACK 只代表网关已经持久化接收，不代表后端已写入成功。

## 重试

导出到后端失败时，网关按路由策略重试。恢复 WAL 或重试超时请求时，下游可能收到已处理过的批次，因此后端或 exporter 需要考虑幂等、去重或可接受重复写入。

## Partial Success

OTLP 的 partial success 不能简单等同于整个请求失败。后续实现 receiver 时，需要将 rejected 记录数量、错误信息和客户端重试行为分开处理，避免把已接收数据重复放大。

## 运维可见性

每条关键路径应暴露队列长度、WAL 写入延迟、WAL 恢复数量、导出成功/失败计数、租户限流计数和策略丢弃计数。
