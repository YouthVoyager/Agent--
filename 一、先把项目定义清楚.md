# 一、先把项目定义清楚

建议把项目暂定为：

> **Policy-first Telemetry Gateway**
> 一个使用 Go 实现的多租户遥测数据网关，在 OTLP、Prometheus、StatsD 与不同观测后端之间执行认证、策略处理、路由、限流、缓冲和回放。

它不是另一个“组件齐全的 OpenTelemetry Collector”，而是专门解决下面的问题：

```text
多个业务团队
   │
   ├── OTLP traces
   ├── OTLP metrics
   ├── OTLP logs
   ├── Prometheus Remote Write
   └── StatsD
          │
          ▼
┌─────────────────────────────┐
│ Policy-first Gateway        │
│                             │
│ 认证 → 租户识别 → 策略      │
│      → 路由 → 队列/WAL      │
│      → 多后端导出           │
└─────────────────────────────┘
          │
    ┌─────┼─────────┐
    ▼     ▼         ▼
  Tempo  Mimir   Datadog/其他
```

OpenTelemetry Collector 本身已经有 receiver、processor、exporter、connector、extension 等成熟组件模型，所以你不应该和它竞争“支持多少组件”。你的差异点应该是：

- 租户是一等公民；
- 策略是一等公民；
- 配置版本、审计、回滚是一等公民；
- 后端迁移和双写是一等公民；
- 可靠性语义明确，而不是“尽量不丢”。

Collector 的 gateway 模式本身也是让应用先发送到统一 OTLP endpoint，再由网关处理并发送到后端，因此你的部署模式与现有 OTel 架构天然兼容。([OpenTelemetry](https://opentelemetry.io/docs/collector/components/?utm_source=chatgpt.com))

# 二、V1 最终实现目标

完成 V1 时，系统应支持下面这个完整场景：

1. A、B 两个团队分别使用 API Key 或 mTLS 接入。
2. 网关将凭证映射到内部 `tenant_id`。
3. 接收 OTLP gRPC 和 OTLP HTTP 的 traces、metrics、logs。
4. 根据 CEL 条件匹配数据。
5. 对属性执行删除、重命名、哈希、脱敏、丢弃、采样。
6. 按租户、服务、信号类型、环境等信息路由。
7. 支持单路发送和多后端 fanout。
8. 后端不可用时进入有界内存队列。
9. 关键路由支持 WAL 持久化。
10. 网关重启后恢复未发送数据。
11. 支持每租户限速、配额和隔离。
12. 支持配置检查、热更新、版本回滚。
13. 暴露自身 metrics、health、readiness、pprof。
14. 支持 Docker Compose 和 Kubernetes 部署。
15. 支持 OTLP metrics 转 Prometheus Remote Write。
16. 支持 Prometheus Remote Write 接入并转换为 OTLP metrics。

暂时不要放进 V1：

- 完整 Prometheus 服务发现；
- 完整 Prometheus scrape scheduler；
- tail-based sampling；
- 完整 Envoy xDS 控制面；
- Web UI；
- Wasm 插件；
- 自研通用查询和存储；
- 几十种 exporter；
- 跨地域共识或分布式 WAL；
- exactly-once。

Prometheus Remote Write 2.0 截至当前官方文档仍标记为 experimental，因此第一版应该默认支持 Remote Write 1.0，并为 2.0 留版本接口，不要直接只实现 2.0。([Prometheus](https://prometheus.io/docs/specs/prw/remote_write_spec_2_0/))

# 三、最重要的架构决定

## 1. 不要自定义一个万能 `map[string]any` 数据模型

内部直接复用：

```go
go.opentelemetry.io/collector/pdata/ptrace
go.opentelemetry.io/collector/pdata/pmetric
go.opentelemetry.io/collector/pdata/plog
go.opentelemetry.io/collector/pdata/pcommon
```

`pdata` 已经提供 traces、metrics、logs 的类型化模型以及 `CopyTo`、`MoveTo` 等操作。它能保留 metric temporality、histogram、resource、scope、trace state 等信息。([Go Packages](https://pkg.go.dev/go.opentelemetry.io/collector/pdata?utm_source=chatgpt.com))

不要设计成：

```go
type Event struct {
    Attributes map[string]any
    Payload    map[string]any
}
```

否则很快会遇到：

- histogram 无法表达；
- cumulative/delta temporality 丢失；
- resource attribute 和 datapoint attribute 混在一起；
- trace event、span link 无法表达；
- Prometheus metadata 无法准确转换；
- 大量 JSON 编解码和内存分配。

推荐定义三种批次：

```go
type Metadata struct {
    TenantID     string
    RequestID    string
    ConfigVersion uint64
    ReceivedAt   time.Time
    Source       string
}

type TraceBatch struct {
    Meta Metadata
    Data ptrace.Traces
}

type MetricBatch struct {
    Meta Metadata
    Data pmetric.Metrics
}

type LogBatch struct {
    Meta Metadata
    Data plog.Logs
}
```

## 2. CEL 只负责匹配，不负责直接修改数据

CEL 适合实现安全、可嵌入、非图灵完备的表达式匹配。([GitHub](https://github.com/google/cel-go?utm_source=chatgpt.com))

例如：

```yaml
match: |
  signal == "logs" &&
  resource["service.name"] == "payment" &&
  attributes["http.response.status_code"] >= 500
```

但是具体 action 应由 Go 实现：

```yaml
actions:
  - delete_attribute:
      key: user.password

  - hash_attribute:
      key: user.email
      algorithm: sha256

  - set_attribute:
      key: data.classification
      value: confidential

  - route:
      destination: security-backend
```

不要允许 CEL 任意构造和修改整个 telemetry payload。这样可以：

- 限制策略复杂度；
- 更好地统计性能；
- 对 action 做静态检查；
- 保证修改语义一致；
- 容易做审计和回放。

## 3. 租户身份必须来自认证结果

不要信任客户端自己发送的：

```text
x-tenant-id: finance
```

正确流程是：

```text
API Key / mTLS certificate
            │
            ▼
    Tenant Resolver
            │
            ▼
       tenant_id
```

例如：

```yaml
tenants:
  finance:
    authentication:
      api_keys:
        - ${FINANCE_API_KEY}

  search:
    authentication:
      mtls:
        allowed_sans:
          - search-prod.example.internal
```

Payload 中即使存在 `tenant.id`，也只能当普通属性处理，不能覆盖认证得到的租户。

## 4. 明确定义 ACK 语义

你需要从一开始明确：

### 非持久模式

```text
收到请求
  → 数据成功放入内存队列
  → 返回成功
```

这代表：

> Gateway 接受了数据，但进程崩溃时队列中的数据可能丢失。

### 持久模式

```text
收到请求
  → 写入 WAL
  → WAL 按配置完成 fsync
  → 返回成功
```

这代表：

> Gateway 已经将数据持久化，但不代表后端已经写入。

不要宣称 exactly-once。实际更合理的目标是：

> **Gateway 内部提供 at-least-once delivery，后端可能收到重复批次。**

OTLP 对可重试错误、partial success 和 retry 行为已有明确规定；特别是 partial success 被填充时，客户端不应简单重试整个请求。([OpenTelemetry](https://opentelemetry.io/docs/specs/otlp/?utm_source=chatgpt.com))

### 目标 2：定义类型化 Pipeline 接口

```go
type TraceConsumer interface {

}
```

### 目标 11：实现 Router 和 Fanout

配置：

```yaml
routes:
  default:
    exporters: [otlp/default]

  security:
    exporters: [otlp/security]

  migration:
    exporters:
      - otlp/old-backend
      - otlp/new-backend
```

关键点：

- 每个 exporter 独立 worker；
- 每个 exporter 独立有界队列；
- 一个 exporter 阻塞不能阻塞另一个；
- fanout 时必须正确复制 pdata；
- 每个 route 有独立 metrics。

验收：

- old backend 停机时，new backend 仍持续接收；
- security route 只收到匹配数据；
- 默认 route 收到未匹配数据；
- 同一条数据不会因为一个分支修改而污染另一个分支。

此时发布：

```text
v0.2.0 — Multi-tenant policy routing
```

## 阶段 D：实现可靠性

### 目标 12：实现有界内存队列

队列必须有界：

```yaml
queue:
  capacity:
    batches: 1000
    bytes: 256MiB
```

同时限制：

- batch 数量；
- 总字节数；
- 单租户占用；
- 单 route 占用。

队列满时支持：

```yaml
overflow_policy: reject
```

第一版只实现 `reject`，不要一开始加入 `drop_oldest`、优先级队列等复杂行为。

验收：

- 队列达到上限后内存不再增长；
- receiver 返回 ResourceExhausted 或 HTTP 429；
- 不产生无限 goroutine；
- queue size、capacity、rejection metrics 正确。

### 目标 14：实现最小 WAL

不要一开始做复杂数据库。自己实现简单 segment WAL，学习价值更高。

记录格式可以是：

```text
| magic |
| version |
| record length |
| sequence number |
| destination id |
| signal type |
| payload |
| crc32 |
```

目录：

```text
wal/
├── 0000000000000001.seg
├── 0000000000000002.seg
└── checkpoint
```

先实现：

- append；
- CRC；
- segment rotation；
- sequence number；
- fsync policy；
- startup scan；
- corrupted tail 截断；
- ack；
- 已完成 segment 删除。

接口：

```go
type WAL interface {
    Append(ctx context.Context, record Record) (Sequence, error)
    ReadFrom(ctx context.Context, seq Sequence) Iterator
    Ack(ctx context.Context, seq Sequence) error
    Close() error
}
```

验收：

1. 写入 1000 条；
2. kill -9；
3. 重启；
4. 恢复未 ack 数据；
5. 已 ack 数据不再发送；
6. 最后一条只写了一半时能够恢复；
7. CRC 错误被检测；
8. 磁盘达到配额后拒绝新数据。

### 目标 15：接入 Durable Queue

数据路径变为：

### 目标 21：实现安全热更新

正确流程：

```text
读取新配置
  → schema 校验
  → CEL 编译
  → exporter 连通性可选检查
```

```go
atomic.Pointer[RuntimeSnapshot]
```

- 无效配置不会影响当前服务；
- 新请求使用新版本；
- 已进入队列的数据继续使用进入时的 route/export destination；
- 每个 batch 记录 `config_version`；
- 可以回滚上一版本。

### 目标 26：故障注入测试

必须覆盖：

| 故障             | 预期行为               |
| ---------------- | ---------------------- |
| 后端 5xx         | 重试并进入队列         |
| 后端 4xx         | 不无限重试             |
| 后端 429         | backoff 或 Retry-After |
| Gateway kill -9  | 重启后 WAL 恢复        |
| WAL 尾部损坏     | 截断损坏尾部           |
| 磁盘满           | 拒绝 durable 请求      |
| 内存队列满       | 返回 backpressure      |
| 配置错误         | 保留旧配置             |
| 单 exporter 卡死 | 不影响其他 exporter    |
| 单租户超额       | 不影响其他租户         |
| DNS 失败         | 有界重试               |
| TLS 证书过期     | 明确失败并告警         |

OTel 的官方可靠性文档同样指出，内存队列满、重试超时、进程崩溃、磁盘满和错误配置都是实际的数据丢失来源，因此这些不能只写单元测试，必须做故障注入。([OpenTelemetry](https://opentelemetry.io/docs/collector/resiliency/))

### 目标 27：性能基准

不要一开始宣称“百万 QPS”。


- bytes/s；
- p50/p95/p99 latency；
- allocs/op；
- bytes allocated/op；
- CPU；
- RSS；
- GC pause；
- queue contention；
- WAL write latency；

测试矩阵：


- exporter 故障时内存不会持续上涨；
- `go test -race` 全部通过。

```yaml
server:
  admin_address: 0.0.0.0:8080

receivers:
  otlp:
    grpc:
      address: 0.0.0.0:4317

      address: 0.0.0.0:4318
      max_request_size: 16MiB

authentication:
  api_key:
    header: Authorization

  finance:
    api_key_hashes:
      - ${FINANCE_KEY_HASH}

    quota:
      bytes_per_second: 20MiB
      max_queue_bytes: 512MiB
      max_wal_bytes: 10GiB

    policies:
      - redact-secrets
      - route-payment-errors

policies:
  redact-secrets:
    rules:
      - match: |
          signal == "logs" &&
          has(attributes["http.request.header.authorization"])
        actions:
          - delete_attribute:
              key: http.request.header.authorization

      - match: |
          signal == "logs" &&
          has(attributes["user.email"])
        actions:
          - hash_attribute:
              key: user.email
              algorithm: sha256

  route-payment-errors:
    rules:
      - match: |
          resource["service.name"] == "payment" &&
          severity_number >= 17
        actions:
          - route:
              destination: security

routes:
  default:
    exporters:
      - otlp/primary

  security:
    exporters:
      - otlp/security
      - otlp/primary

exporters:
  otlp/primary:
    endpoint: primary-otel.example.internal:4317
    tls:
      enabled: true

    queue:
      mode: durable
      capacity_bytes: 2GiB

    retry:
      initial_interval: 1s
      max_interval: 30s
      max_elapsed_time: 30m

  otlp/security:
    endpoint: security-otel.example.internal:4317
    tls:
      enabled: true

storage:
  wal:
    directory: /var/lib/telemetry-gateway/wal
    segment_size: 128MiB
    max_size: 20GiB
    fsync: interval
    fsync_interval: 100ms
```

# 八、最推荐你现在开始的第一个开发闭环

先不要写 WAL，也不要写 Prometheus，更不要写控制面。

第一批只完成下面八件事：

```text
1. 工程骨架和生命周期
2. pdata 类型化批次
3. OTLP gRPC logs receiver
4. debug exporter
5. API Key → tenant 映射
6. 一条 CEL 匹配规则
7. delete_attribute action
8. 一个完整 e2e 测试
```

最终演示：

```text
Go 应用发送 OTLP 日志：

{
  "service.name": "payment",
  "user.email": "alice@example.com",
  "http.request.header.authorization": "Bearer secret"
}

        ↓ Gateway

认证为 finance 租户
删除 authorization
哈希 user.email
路由到 debug exporter

        ↓ 输出

{
  "tenant": "finance",
  "service.name": "payment",
  "user.email": "ff8d9819...",
  "authorization": 不存在
}
```

这个闭环完成后，你已经拥有一个可演示的 **policy-first 多租户 OTLP gateway**。之后增加 traces、metrics、队列、WAL 和协议桥，都是在稳定主链路上逐层扩展，而不是同时启动十几个互相依赖的模块。
